# ADR-006: Dual-Backend Async Replication for Provider-Outage Resilience

**Status:** Proposed
**Date:** 2026-07-20

## Context

ARMOR's entire durability story — the canary (`internal/canary`), the provenance chain (`internal/provenance`), and the Phase 6 restore-verifier (`cmd/restore-verifier`, `internal/restoreverifier`, ADR-004) — proves that **what B2 has is intact and restorable**. None of it protects against B2 not having the object at all: an account suspension, a billing/ToS action, an extended regional outage, or an operator error at the bucket level (e.g. an accidental `DeleteBucket`/lifecycle-rule misfire) are all outside every existing safeguard's blast radius, because every safeguard reads its "known-good" answer from the same single provider it's trying to verify.

`docs/disaster-recovery.md` is explicit about this framing: its TL;DR opens with "the MEK is the single point of failure," and "all operational state lives in B2 — a fresh ARMOR instance with the same config can recover all data" is stated as a *strength*. Both are true, but only conditional on B2 itself being reachable and intact. The document has no section for "B2 account or bucket is gone" because there is currently no recovery path for that scenario — a fresh ARMOR instance with the right MEK recovers nothing if the bucket it points at no longer exists.

`internal/backend/backend.go` already defines a `Backend` interface (`Put`, `Get`, `GetRange`, `Head`, `Delete`, `List`, `Copy`, multipart ops, etc.) specifically so storage providers are swappable — plan.md's "Key Features #4: Pluggable Backend Abstraction" names MinIO, R2, Wasabi, and local-filesystem as anticipated future implementations. As of this ADR, `internal/backend/b2.go` is the only production implementation; the interface's promise of a second backend has never been exercised outside test mocks. This ADR is a **decision to build a second implementation and use it for replication**, not a change to the interface itself.

This is distinct from every corruption incident on record (ADR-002, ADR-003, ADR-005): those were bugs in ARMOR's own encryption/multipart logic that would corrupt data identically regardless of which backend it landed on. A second backend does not protect against ARMOR miscomputing ciphertext. It protects against the *storage provider* being the point of failure — a risk category this project has never addressed, for data plan.md's provenance-chain section itself describes as needing "tamper-evident audit trail[s] for financial/trading data."

## Decision

Add a second `Backend` implementation and use it as an **asynchronous, opt-in mirror** of the primary B2 backend, with the following properties:

1. **Async, non-blocking replication.** After a `PutObject`/`CompleteMultipartUpload` acknowledges success against the primary (B2) backend, the object key is enqueued (in-memory channel, same pattern as the existing manifest write path in `internal/manifest`) for a background writer to copy to the secondary backend. The client-facing write ack is never delayed by the secondary write — this preserves the "ingress is free / fast" property that is core to ARMOR's cost model.
2. **Opt-in per deployment, not global.** Controlled by an env var (e.g. `ARMOR_SECONDARY_BACKEND` + its credentials) that defaults unset/disabled. Buckets carrying data whose loss would be catastrophic (the financial/trading buckets this project keeps building integrity tooling for) opt in; low-value scratch buckets don't pay the extra storage cost.
3. **Secondary backend choice left to the implementation bead**, but the natural first candidate is a **local/off-site filesystem backend** (e.g. an NFS/rsync-reachable path on a different physical host than any B2-dependent cluster) — it is the cheapest to implement against the existing `Backend` interface, requires no new vendor relationship, and, unlike a second S3-compatible cloud provider, has an independent failure domain from "a cloud object-storage API is unreachable or suspended." A second cloud backend (R2, Wasabi) remains a valid future implementation of the same interface if a config-level requirement to survive *simultaneous* cloud provider failures ever emerges.
4. **Health and lag are observable.** The existing canary (`internal/canary`) gains a secondary-backend check alongside its existing single-object and multipart checks: write the canary to both backends, verify both read back correctly, and report replication lag (queue depth / oldest unreplicated object age) on `/armor/canary` and as a Prometheus gauge, following the exact pattern already used for `multipart_healthy`.
5. **Read path is unchanged by default.** Ordinary GetObject continues to serve from B2 via Cloudflare — replication does not touch the zero-egress read path or its cost model. The secondary is a write-time insurance policy, read only during an explicit failover/DR procedure (a new `docs/disaster-recovery.md` section, and a `armor-decrypt`-style direct-to-secondary tool for the "B2 is gone" drill, mirroring how `cmd/restore-verifier`'s DR-drill mode already proves the "ARMOR server is gone" scenario for the primary).
6. **Replication is best-effort at the object level, not transactional.** A window exists between primary-write-ack and secondary-write-complete where only B2 has the object. This is an explicit, documented tradeoff (see Consequences) — closing it would require synchronous dual-write, which is explicitly rejected below.

## Alternatives Considered

1. **Do nothing; rely on B2's own durability + more frequent restore verification.** Rejected: restore-verification (ADR-004) proves *what's in B2 is restorable*, it cannot prove anything if B2 itself is unreachable or the bucket is gone — it shares fate with the exact failure mode it would need to detect.
2. **Synchronous dual-write (block the client ack until both backends confirm).** Rejected: doubles write latency on every PutObject/CompleteMultipartUpload, and a slow or degraded secondary would throttle the primary path — directly undermining "ingress is free" being fast, which is a load-bearing property for ARMOR's DuckDB/litestream write workloads.
3. **Rely on B2's own cross-region replication or lifecycle features instead of an ARMOR-level secondary.** Rejected: still single-provider risk — an account-level suspension or ToS action affects every region under that account equally. Also outside ARMOR's control/observability; ARMOR would have no way to verify it's actually happening.
4. **Full symmetric multi-backend (client reads can be served from either backend, load-balanced).** Rejected as premature: the actual risk being mitigated is *provider outage*, not routine load or latency — symmetric read distribution adds real complexity (cache coherence across two backends' Cloudflare-equivalents, doubled read-path testing surface) for no benefit to the problem this ADR exists to solve. Nothing here blocks revisiting this later.

## Consequences

- **Storage cost roughly doubles for any bucket that opts in** — ARMOR's headline cost model (~$6–7/TB-month) becomes ~$12–14/TB-month plus secondary egress/storage terms for replicated buckets. This must be opt-in (see Decision #2), not a default, to keep ARMOR's core value proposition intact for buckets that don't need it.
- **New operational surface:** secondary backend credentials/config to manage, a replication queue that can back up under sustained write load or secondary-backend slowness, and a new canary check to alert on lag — following the same "self-healing canary" pattern already established, so the operational shape is familiar, not novel.
- **A best-effort replication window exists** (Decision #6) — an object written and acknowledged, then immediately lost from B2 before the async copy completes, is still lost. This must be stated plainly in `docs/disaster-recovery.md` rather than oversold as zero-RPO.
- **Failover is a manual/drilled procedure, not automatic.** Read traffic does not silently move to the secondary on B2 failure; that would require the read-path changes explicitly deferred in Decision #5 / Alternative 4. This ADR buys *recoverability*, not automatic *availability*.
- **The `Backend` interface's design is validated** by finally getting a second real implementation — any latent interface assumptions specific to B2 (e.g. anything not cleanly abstracted in `internal/backend/backend.go`) will surface here rather than staying theoretical.
