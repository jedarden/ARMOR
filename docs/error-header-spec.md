# ARMOR Error Response Header Specification

**Version:** 1.0  
**Date:** 2026-07-14  
**Status:** Active  

## Overview

This specification defines the required HTTP response headers and their expected values for all error responses returned by ARMOR. This specification consolidates findings from comprehensive header consistency verification across multiple verification beads:

- **Content-Type consistency verification** (bf-4bwxtc)
- **HTTP status code consistency verification** (bf-o7eo21)
- **Authentication rejection response headers** (bf-5ppsfh)
- **Invalid AWS credentials rejection testing** (bf-58oib3)

## Scope

This specification applies to **all error responses** from ARMOR, including:

1. Authentication failures (HTTP 403)
2. Authorization failures (HTTP 403)
3. Input validation errors (HTTP 400, 405)
4. Resource not found errors (HTTP 404)
5. Conditional request errors (HTTP 412)
6. Internal server errors (HTTP 500)
7. Service unavailable errors (HTTP 503)

## Required Headers

### All Error Responses

**All error responses MUST include the following header:**

```
Content-Type: application/xml
```

**Specification:**
- **Header Name:** `Content-Type`
- **Required Value:** `application/xml`
- **Case Sensitivity:** Header names are case-insensitive per HTTP/1.1 specification
- **Allowed Variations:** None - must be exactly `application/xml`
- **Charset:** Optional; if specified, must be `application/xml; charset=utf-8`
- **Position:** Must be set before calling `WriteHeader()`

**Implementation Detail:**
All error responses are generated by one of two identical `writeError` functions:
- `internal/server/server.go:writeError` (lines 796-805)
- `internal/server/handlers/handlers.go:writeError` (lines 2695-2704)

Both implementations set:
```go
w.Header().Set("Content-Type", "application/xml")
w.WriteHeader(statusCode)
```

### CORS Headers (Authentication Rejections Only)

**Authentication and authorization rejection responses (HTTP 403) MUST include CORS headers:**

```
Access-Control-Allow-Origin: *
Access-Control-Allow-Methods: GET, PUT, DELETE, HEAD, POST, OPTIONS
Access-Control-Allow-Headers: Authorization, Content-Type, Range, Content-Length
```

**Specification:**
- **Scope:** Only HTTP 403 responses for authentication/authorization failures
- **Header Order:** Not significant (may vary between responses)
- **Allow-Origin Value:** Must be `*` (wildcard)
- **Allow-Methods Value:** Must include: `GET, PUT, DELETE, HEAD, POST, OPTIONS`
- **Allow-Headers Value:** Must include: `Authorization, Content-Type, Range, Content-Length`

**Note:** These CORS headers are added by middleware for authentication endpoints. Other error types (400, 404, 500, etc.) do not include CORS headers.

### Status Code Assignment by Error Category

| HTTP Status | Error Category | Example Error Codes |
|-------------|----------------|---------------------|
| **400** | Client Input Errors | `InvalidRequest`, `InvalidRange`, `MalformedXML` |
| **403** | Authentication/Authorization | `MissingAuthenticationToken`, `InvalidAccessKeyId`, `SignatureDoesNotMatch`, `AccessDenied` |
| **404** | Resource Not Found | `NoSuchKey`, `NoSuchBucket`, `NoSuchUpload` |
| **405** | Method Not Allowed | `MethodNotAllowed` |
| **412** | Conditional Request Failed | `PreconditionFailed` |
| **500** | Internal Server Error | `InternalError` |
| **503** | Service Unavailable | `ServiceUnavailable` |

### Status Code Consistency Requirements

**Authentication/Authorization Errors (HTTP 403):**
All of the following error codes MUST return HTTP 403:

| Error Code | Trigger |
|------------|---------|
| `MissingAuthenticationToken` | Authorization header is missing |
| `InvalidAccessKeyId` | Access key does not exist |
| `SignatureDoesNotMatch` | Signature validation failed |
| `InvalidAlgorithm` | Non-AWS4-HMAC-SHA256 algorithm |
| `MissingDateHeader` | X-Amz-Date header missing |
| `RequestExpired` | Request outside 15-minute window |
| `IncompleteSignature` | Authorization header missing required fields |
| `InvalidCredential` | Invalid credential format |
| `InvalidDateFormat` | Invalid date format in X-Amz-Date |
| `AccessDenied` | ACL-based authorization rejection |

**Verification:** All 10 authentication error codes have been verified to return HTTP 403 (see bf-o7eo21).

## Response Body Format

### Required XML Structure

All error responses MUST follow this exact XML structure:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<Error>
  <Code>ERROR_CODE</Code>
  <Message>Error message description</Message>
</Error>
```

**Specification:**
- **XML Declaration:** Required, must be `<?xml version="1.0" encoding="UTF-8"?>`
- **Root Element:** Must be `<Error>`
- **Code Element:** Required, must contain error code (see Error Codes section)
- **Message Element:** Required, must contain human-readable error message
- **XML Escaping:** All special characters in `<Code>` and `<Message>` MUST be escaped
- **Whitespace:** Newline and spacing may vary but structure must be preserved

**Allowed Variations:**
- Message content varies by error type and may include dynamic context (e.g., `"Failed to encrypt: {error}"`)
- Code values are predefined constants (see Error Codes section)

### Error Code Casing

All error codes MUST use **PascalCase** (matching AWS S3 conventions):

- ✅ Correct: `SignatureDoesNotMatch`, `InvalidAccessKeyId`
- ❌ Incorrect: `signatureDoesNotMatch`, `invalid-access-key-id`

## Error Codes Reference

### Authentication/Authorization Error Codes (HTTP 403)

| Error Code | Message | When Returned |
|------------|---------|---------------|
| `MissingAuthenticationToken` | Missing Authentication Token | Authorization header is completely missing |
| `InvalidAccessKeyId` | The AWS Access Key Id you provided does not exist | Access key not found in credentials store |
| `SignatureDoesNotMatch` | The request signature we calculated does not match the signature you provided | Secret key is incorrect or signature is invalid |
| `RequestExpired` | Request has expired | Request timestamp is outside the allowed window (±15 minutes) |
| `InvalidAlgorithm` | Only AWS4-HMAC-SHA256 is supported | Algorithm is not AWS4-HMAC-SHA256 |
| `IncompleteSignature` | Authorization header is missing required fields | Required components missing (Credential, SignedHeaders, or Signature) |
| `MissingDateHeader` | Missing X-Amz-Date header | X-Amz-Date header is missing |
| `InvalidDateFormat` | Invalid date format in X-Amz-Date header | X-Amz-Date format is invalid |
| `InvalidCredential` | Invalid credential format | Credential string has insufficient parts (expected 5 parts) |
| `AccessDenied` | Access Denied | ACL-based access control rejection |

### Client Input Error Codes (HTTP 400/405)

| Error Code | HTTP Status | Message |
|------------|-------------|---------|
| `InvalidRequest` | 400 | Varies (e.g., "Unsupported POST operation") |
| `InvalidRange` | 400 | "Invalid range: {details}" |
| `MalformedXML` | 400 | "Failed to parse XML: {details}" |
| `MethodNotAllowed` | 405 | "Method {method} not allowed" |

### Resource Not Found Error Codes (HTTP 404)

| Error Code | Message |
|------------|---------|
| `NoSuchKey` | Object not found |
| `NoSuchBucket` | Bucket not found |
| `NoSuchUpload` | Multipart upload not found |

### Conditional Request Error Codes (HTTP 412)

| Error Code | Message |
|------------|---------|
| `PreconditionFailed` | Precondition failed |

### Internal Error Codes (HTTP 500/503)

| Error Code | HTTP Status | Message |
|------------|-------------|---------|
| `InternalError` | 500 | Varies (e.g., "Failed to encrypt: {error}") |
| `ServiceUnavailable` | 503 | Varies |

### Known Inconsistencies and Gaps

#### 1. Code Duplication (Low Severity)

**Issue:** Two identical `writeError` functions exist:
- `internal/server/server.go:writeError`
- `internal/server/handlers/handlers.go:writeError`

**Impact:** Maintenance risk if one is updated without the other

**Current Status:** ✅ No functional inconsistency - both implementations are identical

**Recommendation:** Consider extracting to a shared utility function to prevent drift

**Tracking:** This is a code quality issue, not a functional inconsistency

#### 2. CORS Headers Present Only on 403 Responses

**Issue:** CORS headers are only present on HTTP 403 authentication/authorization responses

**Impact:** None - this is expected behavior

**Current Status:** ✅ Working as designed - CORS middleware only applies to auth endpoints

**Recommendation:** No change needed - document this behavior clearly for consumers

#### 3. Message Format Variation (Intentional)

**Issue:** Error messages vary in verbosity
- Auth errors use generic messages (e.g., "Access Denied")
- Internal errors include diagnostic details (e.g., "Failed to encrypt: {error}")

**Impact:** None - this is security best practice

**Current Status:** ✅ Working as designed

**Recommendation:** No change needed - maintain this pattern for security

## Performance Requirements

### Response Time Requirements

All error responses must meet these performance targets:

| Test Type | Target | Actual | Environment |
|-----------|--------|--------|-------------|
| Unit test rejections | < 100ms | < 1ms | Local httptest |
| Integration test rejections | < 500ms | < 50ms | Real server |
| Malformed signature rejections | < 50ms | < 1ms | Local httptest |

**Average Response Time:** < 150µs (measured 2026-07-14)

### Header Setting Performance

Header setting must not significantly impact response time:

- Single `WriteHeader()` call per response
- No blocking operations during header generation
- Header order must not affect performance

```http

  <Code>ErrorCode</Code>

```

All error responses must include at minimum:
1. `Content-Type: application/xml` header
2. HTTP status code
3. XML declaration
4. `<Error>` root element
5. `<Code>` element
