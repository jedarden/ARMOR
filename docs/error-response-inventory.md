# ARMOR Error Response Inventory

This document provides a comprehensive inventory of all error response scenarios generated by ARMOR handlers, including error codes, categories, HTTP status codes, and test coverage status.

## Error Response Format

All ARMOR error responses follow the S3 XML error format:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<Error>
  <Code>ERROR_CODE</Code>
  <Message>Human-readable error message</Message>
</Error>
```

**Response Headers:**
- `Content-Type: application/xml`

**Performance Target:** All error responses should complete within 100ms

## Error Categories

### 1. Authentication Errors (HTTP 403 Forbidden)

Authentication errors occur when request credentials cannot be verified.

| Error Code | Message | Trigger | Test Coverage |
|------------|---------|---------|---------------|
| `MissingAuthenticationToken` | Missing Authentication Token | Authorization header is missing | ✅ Yes |
| `InvalidAccessKeyId` | The AWS Access Key Id you provided does not exist | Provided access key not found in credentials | ✅ Yes |
| `SignatureDoesNotMatch` | The request signature we calculated does not match the signature you provided | Calculated signature doesn't match provided signature | ✅ Yes |
| `InvalidAlgorithm` | Only AWS4-HMAC-SHA256 is supported | Authorization header doesn't start with "AWS4-HMAC-SHA256" | ✅ Yes |
| `IncompleteSignature` | Authorization header is missing required fields | Auth header missing Credential, SignedHeaders, or Signature | ✅ Yes |
| `MissingDateHeader` | Missing X-Amz-Date header | X-Amz-Date header not provided | ✅ Yes |
| `InvalidDateFormat` | Invalid date format in X-Amz-Date header | Date format not valid "20060102T150405Z" | ⚠️ Limited |
| `RequestExpired` | Request has expired | Request timestamp outside 15-minute window | ✅ Yes |
| `InvalidCredential` | Invalid credential format | Credential doesn't have 5 parts (access/date/region/service/aws4_request) | ⚠️ Limited |

| Precondition | 1 | ⚠️ Limited |

| **Total** | **76+** | **~15%** |

**Overall Assessment:**
- **Strong Coverage:** Authentication and authorization errors are well-tested
- **Weak Coverage:** Validation errors have limited test coverage
- **Missing Coverage:** Internal errors and method-based errors lack dedicated tests
- **Risk:** Most error scenarios lack verification that they return correct HTTP status codes and messages

## Recommendations

1. **Add validation error tests** - Create dedicated tests for all validation error scenarios
2. **Add method-based tests** - Test that unsupported HTTP methods return 405
3. **Add not-found tests** - Test NoSuchKey and NoSuchBucket scenarios with real backend
4. **Document internal error paths** - While deterministic testing is difficult, document the error paths
5. **Add conditional request tests** - Comprehensive testing of If-Match/If-None-Match headers
6. **Performance tests** - Verify all error responses complete within 100ms target
