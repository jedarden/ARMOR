## Test Usage Patterns

### Constructor-based tests (preferred pattern):
```go
err := NewParseError("config.yaml", "invalid syntax", 10, 5, ErrCodeInvalidSyntax, "", "")
err := NewValidationError("config.yaml", "invalid value", "server.port", "must be 1-65535", ErrCodeInvalidValue)
err := NewTypeMismatchError("config.yaml", "server.port", "integer", "string", "abc", 15, "")
err := NewConstraintError("config.yaml", "server.port", "range", "must be 1-65535", "70000", 12, "")
```

### Direct struct construction (currently used for FieldNotFoundError):
```go
// TestFieldNotFoundErrorFormatting (line 648)
err := &FieldNotFoundError{
    FilePath:  "config.yaml",
    FieldPath: "database.host",
    Line:      8,
}
```

## Key Findings

1. **Missing Constructors**: 4 error types lack constructors (SyntaxError, StructureError, DuplicateKeyError, SchemaLoadError)

2. **Inconsistent Test Usage**: `FieldNotFoundError` has a constructor (`NewFieldNotFoundError()` at line 494) but tests use direct struct construction instead

3. **No Test Coverage**: Error types without constructors also have no test coverage

4. **Constructor Pattern**: Existing constructors follow consistent patterns:
   - Accept required parameters first (filePath, message, etc.)
   - Accept optional parameters last (line, column, errorCode)
   - Provide default error codes if empty string passed
   - Return properly initialized struct pointers

## Impact on Tests

- Tests currently use direct struct construction for `FieldNotFoundError` despite having a constructor
- No tests exist for the 4 error types lacking constructors
- To add comprehensive test coverage, constructors should be added first to maintain consistency
