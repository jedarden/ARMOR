## 1. test_valid_complete_yaml

**Error:**
```
DEBUG - Found 2 errors in valid YAML:
  0 path=key_host line=Some(4) msg=duplicate key 'host' appears 2 times in document
  1 path=key_port line=Some(5) msg=duplicate key 'port' appears 2 times in document
```

**YAML Content:**
```yaml
# Configuration file
server:
  host: localhost
  port: 8080
database:
  host: db.example.com
  port: 5432
  name: mydb
features:
  - authentication
  - logging
  - caching
env:
  production: true
  debug: false
```

**Root Cause:**
The global duplicate key detection in `finalize_structure_checks()` (syntax_detector.rs:743-757) is incorrectly flagging keys that appear in **different mappings** as duplicates.

Keys `host` and `port` appear in both the `server:` mapping and the `database:` mapping, which is perfectly valid YAML. Keys only need to be unique within their parent mapping scope, not globally across the entire document.

## Bug Location

**File:** `src/parsers/yaml/syntax_detector.rs`
**Function:** `finalize_structure_checks()` (lines 743-757)
**Code:**
```rust
fn finalize_structure_checks(&mut self, errors: &mut Vec<ValidationError>) {
    // Check for global duplicate keys
    for (key, line_nums) in &self.structure_state.all_keys {
        if line_nums.len() > 1 {
            // Only report if not at same indentation level (already reported)
            let is_same_level = line_nums.windows(2).all(|w| w[0] == w[1]);
            if !is_same_level {
                errors.push(ValidationError::new(
                    format!("key_{}", key),
                    format!("duplicate key '{}' appears {} times in document", key, line_nums.len())
                ).with_line(line_nums[0]));
            }
        }
    }
}
```

## Root Cause Analysis

The bug is in the logic that attempts to distinguish between:
1. **True duplicates** - same key appearing multiple times at the same indentation level (within the same mapping)
2. **False positives** - same key name appearing in different mappings (valid YAML)

The problematic code:
```rust
let is_same_level = line_nums.windows(2).all(|w| w[0] == w[1]);
```

This compares **line numbers** (`w[0] == w[1]`), not indentation levels. Since different key occurrences will always be on different lines, this condition is almost always false, causing the error to be reported even when the keys are in different mappings.

The real issue is that `all_keys` stores only line numbers, not the indentation context needed to determine if keys are in the same mapping scope.

## Impact

This is a **false positive bug** - the detector reports duplicate keys where none exist, causing valid YAML files to be rejected. This is particularly problematic for configuration files that commonly use the same key names (like `host`, `port`, `name`) in different sections.

## Recommended Fix

The global duplicate key detection should either:
1. Be removed entirely (same-level duplicate detection is sufficient)
2. Track indentation levels along with keys to properly scope duplicate detection
3. Only report global duplicates when the same key path is duplicated (e.g., `services.web.host` vs `services.database.host` are different paths)

The `current_keys` tracking (lines 678, 701-702) already correctly handles same-level duplicate detection by clearing when a new parent key is encountered. The global detection adds no value and causes false positives.
