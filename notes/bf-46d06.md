## The 3 Previously Failing Tests

### 1. TestFieldNotFoundErrorFormatting (bf-3xnxx)
**File:** `internal/yamlutil/errors_test.go:671`
**Purpose:** Test formatting of FieldNotFoundError messages with different scenarios
**Status:** ✅ Fixed in commit `60237956`

**Changes Applied:**
- Replaced direct `&FieldNotFoundError{}` struct construction with `NewFieldNotFoundError()` constructor calls
- Updated 3 test cases:
  - "field not found with line number"
  - "field not found without line number"  
  - "field not found with nested field path"

**Example Fix:**
```go
// Before:
err: &FieldNotFoundError{
    FilePath:  "config.yaml",
    FieldPath: "database.host",
    Line:      8,
}

// After:
err: NewFieldNotFoundError("config.yaml", "database.host", 8, "")
```

### 2. TestValidationErrorString (bf-1qpjm)

- Updated 3 test cases:

  - "validation error without constraint"
  - "validation error with line and column"

## Error Constructor Fixes Applied

### Pattern of Fixes
All three fixes follow the same pattern:
1. **Identify direct struct construction** - Tests using `&ErrorType{Field: value}` pattern
2. **Replace with constructor** - Use `NewErrorType(params)` instead
3. **Preserve test logic** - Keep identical test data and assertions
4. **Ensure consistency** - All error types now use constructor pattern

### Benefits of Constructor Pattern
- **Centralized initialization** - All error types initialized consistently
- **Validation logic** - Constructors can validate parameters
- **Future-proofing** - Easy to add fields or validation later
- **Test coverage** - Constructor functions themselves are tested
