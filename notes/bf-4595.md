## Implementation Details

### 1. Canary Monitor Extensions (`internal/canary/canary.go`)

**New State Fields:**
- `MultipartHealthy` - Health status of multipart canary (healthy/unhealthy/unknown)
- `MultipartLastCheck` - Timestamp of last multipart check
- `MultipartLastSuccess` - Timestamp of last successful multipart check
- `MultipartConsecutiveFails` - Counter for consecutive multipart failures
- `MultipartLastError` - Error message from last failed multipart check

**New Configuration:**
- `MultipartInterval` - Check interval (default: 1 hour)
- `MultipartSize` - Size of multipart canary object (default: 6MB)

**Core Methods:**
- `checkMultipart()` - Performs multipart canary integrity check:
  - Generates 6MB random canary content (above 5MB multipart threshold)
  - Encrypts content using ARMOR envelope encryption
  - Uploads via real multipart S3 API:
    - `CreateMultipartUpload()` - Initiates multipart upload
    - `UploadPart()` - Uploads 2MB parts
    - `CompleteMultipartUpload()` - Finalizes upload
  - Downloads and verifies:
    - Byte-for-byte comparison
    - HMAC verification
    - Plaintext SHA-256 verification
    - Decryption verification
  - Cleans up canary object (async best-effort)

- `runMultipartCheck()` - Executes multipart check with retry logic (max 3 retries)
- `updateMultipartStateSuccess()` - Updates state on successful check
- `updateMultipartStateFailure()` - Updates state on failed check

**Scheduling:**
- Regular canary checks run every 5 minutes (small object, single-part upload)
- Multipart canary checks run every 1 hour (large object, multipart upload)
- Both checks run independently with separate tickers

### 2. Metrics Integration (`internal/metrics/metrics.go`)

**New Metrics:**
- `armor_multipart_canary_checks_total` - Counter for total multipart checks

## Key Design Decisions

1. **6MB Multipart Size** - Above 5MB multipart threshold to ensure multipart API is used, but small enough for quick checks

2. **1 Hour Interval** - Balances thoroughness with cost (multipart operations are more expensive than single-part)

3. **Independent Status Tracking** - Multipart health is tracked separately from regular canary so a multipart-specific regression is visible even if small-object checks remain green

4. **Same Verification Logic** - Reuses existing HMAC/SHA/byte-for-byte verification code to ensure consistency

5. **2MB Part Size** - Reasonable part count (3 parts for 6MB object) for testing the multipart path without excessive overhead
