# bf-42c149: Starvation alert — beads invisible to worker

## Task Description
NEEDLE starvation alert: "Open beads exist but Pluck found none — possible
configuration error." Workspace `/home/coding/ARMOR`, 1602 beads, 2 open,
0 in-progress, 0 claimed at snapshot time. Asked to check `exclude_labels`,
workspace path, and filter configuration.

## Finding
**Not a configuration error. The starvation is genuine and expected.** Pluck
correctly found no ready beads because the only two open beads are both
dependency-blocked Genesis (root) beads, so `br ready` returns the empty set.

### Config check (all clean — the alert's suggested checklist is a red herring here)
- `exclude_labels: []` in `.needle.yaml` → no label-based exclusions.
- Workspace path correct — `.beads/` in `/home/coding/ARMOR`.
- No filter misconfiguration; `br count` = 1602, `br list --status open` = 2.

### Why no bead is ready
Both open beads are Genesis beads whose remaining children are themselves
blocked:

**`bf-42dng8`** — *Genesis: ARMOR Phase 6 Backup Restore Verification* (open, P1)
- 4 of 5 blockers closed (`bf-1tzyle`, `bf-1ngen7`, `bf-p607oj`, `bf-1pphhz`).
- 1 still blocked: **`bf-1ebnuz`** — *Multipart-era corruption audit of
  unaudited ARMOR buckets*. Description is explicitly
  **CREDENTIAL-GATED: "do not dispatch to unattended workers (requires live
  B2/cluster credentials an operator must supply)."** It carries zero
  dependencies — `blocked` is a manual operator hold, not an unmet dep.

**`bf-4l7q`** — *Genesis: ARMOR Integrity Detection Hardening (Phase 5)* (open, P1)
- 3 of 6 blockers closed (`bf-4595`, `bf-28rb`, `bf-16f6`).
- 3 still blocked:
  - `bf-2t1f` (P2, version-drift check) — both its deps (`bf-1ia5y1`,
    `bf-3x13d8`) are closed, but it is manually held `blocked`.
  - `bf-4qq1` (P0, bump ord-devimprint ARMOR + verify restore) — blocked by
    `bf-5aqh0` (test-restore queue-api), which is itself genuinely blocked.
  - `bf-1v6skf` (P0, large-multipart HMAC failure) — **all 4 of its deps are
    closed** (`bf-28rb`, `bf-4595`, `bf-24sxh7`, `bf-2sq7gf`), yet it is
    manually held `blocked`.

### Secondary observation (bead-forge behavior, not acted on)
`blocked` is a **manual operator status**, independent of dependency
resolution: `bf-1ebnuz` and `bf-1v6skf`/`bf-2t1f` are blocked with all (or
no) dependencies closed. bead-forge does **not** auto-clear a manual `blocked`
flag when a bead's dependency chain resolves. This is by design — the flags
represent deliberate operator gating (credential/ops holds) — so no state was
mutated here. If the operator wants to surface that work, they can `br reopen`
`bf-1v6skf` / `bf-2t1f` manually; unattended workers should not.
