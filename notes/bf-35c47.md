## Investigation Summary

The bead description referenced "undefined field references (filePath0, filePath1, etc.)" but the actual issues found were different:

### Issue #1: Missing Function Parameters (RESOLVED in commit fdf4871f)

**Root Cause:** The `NewValidationError` function signature requires 11 parameters, but several test calls were only providing 9 parameters.

**Function Signature:**
```go
func NewValidationError(
    filePath string,
    message string,
    fieldPath string,
    constraint string,
    code ErrorCode,
    line int,
    column int,
    errorType ErrorType,
    path string,
    expectedType string,  // ← These were missing
    actualType string     // ← These were missing
) *ValidationError
```

### Issue #2: Direct Field Access (Architectural Issue)

**Status:** Documented in bead bf-10u6n

The tests directly access internal struct fields, which violates the encapsulation pattern. However, this is an architectural issue, not undefined field references.

**Examples:**
```go
if err.FilePath != tt.filePath {        // Direct field access
    t.Errorf("FilePath = %q, want %q", err.FilePath, tt.filePath)
}
```

**Fields accessed:**
- `err.FilePath`, `err.Message`, `err.Line`, `err.Column`, `err.Expected`, `err.Actual` (ParseError)
- `err.FilePath`, `err.Message`, `err.FieldPath`, `err.Constraint`, `err.Line`, `err.Column` (ValidationError)

All these fields **exist** on the structs - the issue is architectural (coupling tests to implementation), not syntactic.

## Conclusion

### No Undefined Field References Found
The search for patterns like `filePath0`, `filePath1`, etc. found **NO instances**. The bead description used these as examples, but they don't exist in the codebase.

### Actual Issue: Missing Parameters
The real problem was that `NewValidationError` calls were missing the `expectedType` and `actualType` parameters. This was fixed in commit fdf4871f.
