# Test Verification: Exclamation Mark Handling

## Executive Summary (2026-07-13 - Updated)

⚠️ **Exclamation mark handling has ISSUES in block scalar contexts**

- **258 of 262 tests pass** (98.5% pass rate)
- **4 test failures ARE related to exclamation mark handling** in block scalar contexts
- Failures stem from a **design limitation**: `classify_line_type()` lacks multiline context for block scalar continuation lines

**Clarification:** While basic exclamation mark handling (in comments, values, tags) works correctly, the parser fails to properly handle exclamation marks in block scalar continuation lines (folded `>` and literal `|` styles). This affects lines like `  !start and end!` which appear in literal block content.

### Test Summary

- **Passed:** 258

### Failures Found

#### 1. `test_detect_mapping_key_sequence_items_rejected`
**Expected:** Sequence item `- !ns:tag` should return `None` from `detect_mapping_key`

**Actual:** Being detected as a mapping key when it shouldn't be

**Root Cause:** The line starts with `-`, so it's classified as `SequenceItem`, but the test expects `detect_mapping_key` to return `None` for sequence items.

#### 2. `test_folded_style_scalars_with_exclamation`
**Expected:** Folded scalar continuation line `'  This is important! Read carefully.'` should be `Unknown` or `Tag`

**Actual:** Being classified as `MappingKey`

**Root Cause:** Line doesn't contain `:` and falls through to default `LineType::MappingKey` at line 729 in `classify_line_type`.

#### 3. `test_literal_style_scalars_with_exclamation`
**Expected:** Line `'  !start and end!'` should be `MappingKey` or `Comment`

**Actual:** Failing classification

**Root Cause:** Similar to #2 - continuation lines in block scalars are being misclassified.

#### 4. `test_multiline_comment_and_config_mixed_with_exclamation`
**Expected:** Line `'  This is a multiline'` should be `Unknown`

**Actual:** Being classified as `MappingKey`

**Root Cause:** Same underlying issue - continuation lines after block scalar markers fall through to default `MappingKey`.

### Root Cause Analysis

The core issue is in `src/parsers/yaml/line_parser.rs` in the `classify_line_type` function (lines 727-729):

```rust
// Default to mapping key for lines without special patterns
// This handles literal content lines that may contain ! or other characters
LineType::MappingKey
```

The function falls through to `MappingKey` for any line that doesn't match earlier patterns, but **indented continuation lines in block scalars** should be `Unknown`.

### Recommendation

The `classify_line_type` function needs to track context (whether inside a block scalar) or use indentation to determine continuation lines.

- Actual behavior: `classify_line_type()` returns `LineType::Unknown`
- Root cause: Line does not contain `:` (colon) and does not start with `#`, so it falls through to default `Unknown` classification
- **Assessment:** Test expectation appears incorrect. This is a content line, not a mapping key or comment.

#### 2. `test_multiline_yaml_strings_with_exclamation_in_nested_contexts`  
**Line:** 6954
**Panic message:** `Should detect mapping key in nested multiline: '  - name: item1'`

**Analysis:**
- Test expects `'  - name: item1'` to be detected as a mapping key
- Actual behavior: Not being detected as expected
- Root cause: This is a sequence item (starts with `- `), not a direct mapping key
- **Assessment:** Test expectation may need adjustment for sequence item handling.

### Classification Logic

The `classify_line_type()` function in `src/parsers/yaml/line_parser.rs` follows this logic:

1. Lines starting with `#` → `Comment`
2. Lines containing `:` → `MappingKey`
3. Lines starting with `-` → `SequenceItem`
4. All other content → `Unknown`

The failing tests expect content lines without colons (like `'  echo 'Done! Complete!''`) to be classified as `MappingKey` or `Comment`, which contradicts the design of the classifier.

### Exclamation Mark Handling Status

Exclamation marks (`!`) are handled correctly by the parser:
- Lines starting with `!` are classified as `Tag` (line 681-683)
- Exclamation marks elsewhere in the line do not trigger special classification
- This is the correct behavior per YAML specification

### Conclusion

The test failures are due to **incorrect test expectations**, not bugs in exclamation mark handling:
1. Content lines without colons should be `Unknown`, not `MappingKey` or `Comment`
2. Sequence items (`- key: value`) should be handled as `SequenceItem`, not direct mapping keys

The tests appear to have been written without fully considering the classification rules. The actual exclamation mark handling is working as designed.

### Recommendation

The test expectations in `test_literal_style_scalars_with_exclamation` and `test_multiline_yaml_strings_with_exclamation_in_nested_contexts` should be revised to match the actual (correct) behavior of the line classifier.
