## Detailed Analysis

### ✅ Category 1: S3 API Authentication/Authorization Rejections

**Status: PERFECTLY CONSISTENT**

All authentication and authorization errors use the `writeError()` function (`server.go:796-805`).

#### Response Headers (identical for all 10 scenarios)
```
Content-Type: application/xml
Access-Control-Allow-Origin: *
Access-Control-Allow-Methods: GET, PUT, DELETE, HEAD, POST, OPTIONS
Access-Control-Allow-Headers: Authorization, Content-Type, Range, Content-Length
```

#### Status Code
`403 Forbidden` (all scenarios)

#### Response Body Format
```xml
<?xml version="1.0" encoding="UTF-8"?>
<Error>
  <Code><ErrorCode></Code>
  <Message><ErrorMessage></Message>
</Error>
```

#### Rejection Types (all verified)

| Error Code | Trigger Condition | Message |
|------------|------------------|---------|
| MissingAuthenticationToken | No Authorization header | Missing Authentication Token |
| InvalidAccessKeyId | Unknown access key | The AWS Access Key Id you provided does not exist |
| SignatureDoesNotMatch | Wrong signature | The request signature we calculated does not match the signature you provided |
| InvalidAlgorithm | Wrong auth algorithm | Only AWS4-HMAC-SHA256 is supported |
| IncompleteSignature | Empty/malformed signature | Authorization header is missing required fields |
| MissingDateHeader | No X-Amz-Date header | Missing X-Amz-Date header |
| InvalidDateFormat | Malformed date | Invalid date format in X-Amz-Date header |
| RequestExpired | Timestamp outside ±15min window | Request has expired |
| InvalidCredential | Malformed credential string | Invalid credential format |
| AccessDenied | ACL-based rejection | Access Denied |

#### Test Coverage
- ✅ `TestContentTypeConsistencyAcrossAllRejections` - All 10 scenarios verified
- ✅ `TestComprehensiveErrorVerification` - Performance < 100ms verified (actual: 3-15µs)
- ✅ `TestAuthRejectionHeadersDocumentation` - Headers documented

#### Implementation

**Error Writer:** `internal/server/server.go:796-805`
```go
func (s *Server) writeError(w http.ResponseWriter, code, message string, statusCode int) {
    w.Header().Set("Content-Type", "application/xml")
    w.WriteHeader(statusCode)
    var codeBuf, msgBuf bytes.Buffer
    xml.EscapeText(&codeBuf, []byte(code))
    xml.EscapeText(&msgBuf, []byte(message))
    fmt.Fprintf(w, `<?xml version="1.0" encoding="UTF-8"?>`+"\n<Error>\n  <Code>%s</Code>\n  <Message>%s</Message>\n</Error>",
        codeBuf.String(), msgBuf.String())
}
```

**CORS Middleware:** `internal/server/server.go:656-659`
```go
// Add CORS headers for browser clients
w.Header().Set("Access-Control-Allow-Origin", "*")
w.Header().Set("Access-Control-Allow-Methods", "GET, PUT, DELETE, HEAD, POST, OPTIONS")
w.Header().Set("Access-Control-Allow-Headers", "Authorization, Content-Type, Range, Content-Length")
```

The CORS headers are added by `wrapHandler` middleware BEFORE calling the handler, ensuring all S3 API responses (including errors) include them.

### ⚠️ Category 2: Admin/Management API Errors

**Status: INTENTIONAL INCONSISTENCY**

Admin endpoints (`/admin/*`) use `http.Error()` directly, returning plain text without CORS headers.

#### Response Headers
```
Content-Type: text/plain (default)
```

**No CORS headers are set**

#### Examples
| Endpoint | Error | Status | Format |
|----------|-------|--------|--------|
| `/admin/key/verify` | Method not allowed | 405 | Plain text |
| `/admin/key/rotate` | Failed to read body | 400 | Plain text |
| `/admin/key/export` | Must include ?confirm=yes | 400 | Plain text |
| `/admin/b2/keys` | B2 key management not available | 503 | JSON string |

**Rationale:** Admin endpoints are internal operations, not S3 protocol. Plain text is acceptable for human-readable errors.

### ✅ Verification Complete

### ⚠️ Intentional Differences

The header differences between API categories are **intentional and correct**:

- **S3 API** → XML format with CORS (S3 protocol requirement)
- **Admin API** → Plain text without CORS (internal operations)
- **Share API** → Plain text without CORS (public utility)
- **Health API** → Plain text without CORS (Kubernetes probes)

### No Issues Found

No actual inconsistencies were discovered. The header differences between categories are:
1. **By design** - different categories serve different purposes
2. **Well-tested** - comprehensive test coverage exists
3. **S3-compliant** - S3 protocol requires XML errors for client compatibility
