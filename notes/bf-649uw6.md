# Error Response Header Consistency Analysis

## Task: Verify error response header consistency across all rejection types

### Summary
ARMOR has **inconsistent error response headers** across different API endpoints. While S3 operations maintain consistent XML responses with proper headers, admin endpoints and share endpoints use plain text responses via `http.Error()`.

## Findings

### 1. S3 API Operations (CONSISTENT ✅)

**Location**: `internal/server/handlers/handlers.go:2696`

All S3 operations use the centralized `writeError()` function which sets:
- **Content-Type**: `application/xml`
- **Response Format**: XML S3 error structure
- **Status Codes**: Proper HTTP status codes

```go
func (h *Handlers) writeError(w http.ResponseWriter, code, message string, statusCode int) {
    w.Header().Set("Content-Type", "application/xml")
    w.WriteHeader(statusCode)
    var codeBuf, msgBuf bytes.Buffer
    xml.EscapeText(&codeBuf, []byte(code))
    xml.EscapeText(&msgBuf, []byte(message))
    fmt.Fprintf(w, `<?xml version="1.0" encoding="UTF-8"?>`+"\n<Error>\n  <Code>%s</Code>\n  <Message>%s</Message>\n</Error>",
        codeBuf.String(), msgBuf.String())
}
```

**Rejection Scenarios Covered**:
- Missing authentication header → `MissingAuthenticationToken`
- Invalid access key → `InvalidAccessKeyId`  
- Signature mismatch → `SignatureDoesNotMatch`
- Malformed authorization → `InvalidAlgorithm`
- Missing date header → `MissingDateHeader`
- Expired request → `RequestExpired`
- Empty signature → `IncompleteSignature`
- Invalid signature characters → `SignatureDoesNotMatch`
- ACL denial → `AccessDenied`

### 2. Server Wrapper Authentication (CONSISTENT ✅)

**Location**: `internal/server/server.go:797`

The server's authentication wrapper uses the same pattern:
- **Content-Type**: `application/xml`
- **Response Format**: XML S3 error structure

### 3. Admin API Endpoints (INCONSISTENT ❌)

**Location**: `internal/server/server.go`

Admin endpoints use plain text responses via `http.Error()`:
- **Content-Type**: `text/plain` (default from `http.Error`)
- **Response Format**: Plain text strings
- **Status Codes**: Various HTTP status codes

**Examples**:
```go
// verifyKey, rotateKey, exportKey, canaryHandler, audit
http.Error(w, "Method not allowed", http.StatusMethodNotAllowed)
http.Error(w, fmt.Sprintf("Failed to read request body: %v", err), http.StatusBadRequest)
```

### 4. Share Endpoint (INCONSISTENT ❌)

**Location**: `internal/server/server.go:915-942`

The share endpoint uses plain text responses:
```go
http.Error(w, "Method not allowed", http.StatusMethodNotAllowed)
http.Error(w, "Missing token", http.StatusBadRequest)
http.Error(w, "Link expired", http.StatusGone)
http.Error(w, "Invalid link", http.StatusForbidden)
http.Error(w, "Invalid token", http.StatusBadRequest)
```

## Header Inconsistency Summary

| Endpoint Type | Content-Type | Response Format | Consistency |
|--------------|--------------|----------------|-------------|
| S3 Operations | `application/xml` | XML S3 errors | ✅ Consistent |
| Auth Wrapper | `application/xml` | XML S3 errors | ✅ Consistent |
| Admin API | `text/plain` or `application/json` | Plain text or JSON | ❌ Inconsistent |
| Share API | `text/plain` | Plain text | ❌ Inconsistent |
| B2 Keys API | `text/plain` | JSON in plain text | ❌ Inconsistent |

## Rejection Scenarios Documented

### Authentication Rejections
1. **Missing Authentication Header** - HTTP 403, `MissingAuthenticationToken`
2. **Invalid Access Key** - HTTP 403, `InvalidAccessKeyId`
3. **Signature Mismatch** - HTTP 403, `SignatureDoesNotMatch`
4. **Malformed Authorization** - HTTP 403, `InvalidAlgorithm`
5. **Missing Date Header** - HTTP 403, `MissingDateHeader`
6. **Request Expired** - HTTP 403, `RequestExpired`
7. **Empty Signature** - HTTP 403, `IncompleteSignature`
8. **Invalid Signature Characters** - HTTP 403, `SignatureDoesNotMatch`

### Authorization Rejections
9. **ACL Denial** - HTTP 403, `AccessDenied`

### Operational Rejections
10. **Method Not Allowed** - HTTP 405, `MethodNotAllowed`
11. **Invalid Request** - HTTP 400, various codes
12. **Object Not Found** - HTTP 404, `NoSuchKey`
13. **Internal Error** - HTTP 500, `InternalError`

## Conclusion

The S3 API operations maintain **excellent header consistency** with proper XML error responses. However, admin and utility endpoints show **inconsistent headers** across different response types.

For S3-compatible operations (the primary API surface), header consistency is **verified and consistent** ✅

For auxiliary endpoints (admin, share, B2 keys), header consistency **needs improvement** ❌
