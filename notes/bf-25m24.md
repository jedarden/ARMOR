# Direct Field Access Issues in result_test.go

## Summary

After reviewing bf-558ti error construction patterns, I found **field assignment after construction** patterns in `internal/yamlutil/result_test.go` that violate the established patterns. Reading fields for test verification is acceptable, but modifying fields after construction is not.

## Context from Related Beads

- **bf-337qg**: Already fixed direct field access in result_test.go (line 485) - replaced `err.ContextStr = "initial"` with constructor parameter
- **bf-300fh**: Already fixed direct field access in errors_test.go (lines 796-797) - replaced `err.ExpectedType = tt.expectedType` with constructor parameters
- **bf-558ti**: Established that direct field assignment after construction is prohibited

## Issues Found (Field Assignment After Construction)

### 1. Direct Message field modification (Lines 139, 152)
```go
// Line 139 - TestResult_MapErr
mapped := result.MapErr(func(e *ParseError) *ParseError {
    e.Message = "modified: " + e.Message  // ❌ Direct field assignment
    return e
})

// Line 152 - Same pattern
mappedErr := errResult.MapErr(func(e *ParseError) *ParseError {
    e.Message = "modified: " + e.Message  // ❌ Direct field assignment
    return e
})
```

**Problem**: The MapErr callback modifies error fields directly instead of creating new errors via constructor.

### 2. Direct Err field assignment (Line 523)
```go
// Line 523 - ExampleResult
parseErr := NewParseError("", fmt.Sprintf("invalid number: %s", s), 0, 0, "", "", "")
parseErr.Err = err  // ❌ Direct field assignment after construction
return Err[int, *ParseError](parseErr)
```

**Problem**: Creates error then modifies the Err field directly instead of using proper error wrapping.

## Fix Plan

### Issue 1: MapErr field modification (Lines 139, 152)

**Current problematic code:**
```go
mapped := result.MapErr(func(e *ParseError) *ParseError {
    e.Message = "modified: " + e.Message  // ❌ Direct field assignment
    return e
})
```

**Fix options:**

**Option A:** Modify MapErr to accept a constructor-style function that creates new errors:
```go
mapped := result.MapErr(func(e *ParseError) *ParseError {
    return NewParseError(
        e.FilePath,
        "modified: " + e.Message,
        e.Line,
        e.Column,
        e.ErrorCode,
        e.Expected,
        e.Actual,
        e.ContextStr,
    )
})
```

**Option B:** Add a `WithMessage()` method to ParseError that returns a new error:
```go
func (pe *ParseError) WithMessage(newMsg string) *ParseError {
    return &ParseError{
        FilePath:   pe.FilePath,
        Line:       pe.Line,
        Column:     pe.Column,
        Message:    newMsg,
        ContextStr: pe.ContextStr,
        Err:        pe.Err,
        ErrorType:  pe.ErrorType,
        ErrorCode:  pe.ErrorCode,
        Expected:   pe.Expected,
        Actual:     pe.Actual,
    }
}

// Then in test:
mapped := result.MapErr(func(e *ParseError) *ParseError {
    return e.WithMessage("modified: " + e.Message)
})
```

**Recommendation:** Option B is cleaner for tests and follows immutable error patterns.

### Issue 2: Err field assignment in ExampleResult (Line 523)

**Current problematic code:**
```go
parseErr := NewParseError("", fmt.Sprintf("invalid number: %s", s), 0, 0, "", "", "")
parseErr.Err = err  // ❌ Direct field assignment
```

**Analysis:** `NewParseError` signature does NOT include error wrapping:
```go
func NewParseError(filePath string, message string, line int, column int, code ErrorCode, expected string, actual string, contextStr string) *ParseError
```

**Fix options:**

**Option A:** Extend NewParseError signature to accept error wrapping:
```go
func NewParseError(filePath string, message string, line int, column int, code ErrorCode, expected string, actual string, contextStr string, err error) *ParseError
```

**Option B:** Add a `WrapError()` method to ParseError:
```go
func (pe *ParseError) WrapError(wrapped error) *ParseError {
    return &ParseError{
        FilePath:   pe.FilePath,
        Line:       pe.Line,
        Column:     pe.Column,
        Message:    pe.Message,
        ContextStr: pe.ContextStr,
        Err:        wrapped,
        ErrorType:  pe.ErrorType,
        ErrorCode:  pe.ErrorCode,
        Expected:   pe.Expected,
        Actual:     pe.Actual,
    }
}

// Then in example:
parseErr := NewParseError("", fmt.Sprintf("invalid number: %s", s), 0, 0, "", "", "").WrapError(err)
```

**Option C:** Simplify the example to avoid error wrapping (least intrusive):
```go
// In the example, just return a simple error without wrapping
parseErr := NewParseError("", fmt.Sprintf("invalid number: %s", s), 0, 0, "", "", "")
return Err[int, *ParseError](parseErr)
```

**Recommendation:** Option B (WrapError method) is best for maintaining proper error wrapping while following immutable patterns. Option C is acceptable if the example's error wrapping isn't critical to demonstrate.
